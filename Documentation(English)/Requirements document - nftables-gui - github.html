1. Introduction
The nftables-gui software aims to model the graphical interface of nftables in a
to make it easier to use and less time consuming to implement change iptables
by nftables.
3
Page 4
2. Reach
Nftables-gui aims to offer the functionality developed so far by
nftables, for which the following functionalities can be distinguished:
Table creation / deletion
String creation / deletion
Create / delete rules
Export / import rules using json and xml

3. Introduction to the problem domain
Studying the other project documents, different
processes, then we proceed to explain in more detail how
These processes will be managed:
Create a table:
• Create a table within the nftables system.
Deleting a table:
• Delete a table within the nftables system.
Creating a chain:
• Create a string within the nftables system.
Deleting a string:
• Delete a string within the nftables system.
Create a rule:
• Create a rule within the nftables system.
Delete a rule:
• Delete a rule within the nftables system.
Export rules in xml or json:
• It is intended to export a set of rules to an external file
towards nftables
Import rules in xml or json:
• It is intended to import a set of rules into nftables by
an external file.

4. Glossary of terms
The terms used in this document are explained below:
Table: A table is an object that contains a set of strings, they can be
of type ip, ip6, arp and nat.
String: A string is an object that can (as in iptables) contain a
set of rules, can be of the following types (hook): INPUT, OUTPUT, PREROUTE, POSTROUTE
and CUSTOM.
Rule: A rule is a series of filtering orders on a certain type
package.
Port: A port is an identifying number through which the different
Some of the famous network protocols are: 80 (http) 443 (https), 21 (ftp), 22 (ssh).
IP: An IP is a 32-bit address divided into 8-bit blocks of the type:
xxx.xxx.xxx.xxx.

5. Specification of requirements
5.1. General requirements
The software system can be used by any user with permission
administrator and must store in a database all the information
relevant to the system, being this: tables, chains, rules. On the other hand the
system shall allow a table, chain or rule to be added, modified
modified or eliminated at any time and any user can do it
with permissions.
5.2. Information requirements
RI-001 TABLE: The system must store the corresponding information
to a table, specifically: table name, table family (ip, ip6, arp,
nat, raw, etc) set of strings you have (if you have them) and associated rules
these.
RI-002 CHAIN: The system must store the corresponding information
tooth to a chain, specifically: name the chain, type of chain (INPUT,
OUTPUT, PREROUTE, POSTROUTING, CUSTOM) and the rules associated with
is.
RI-003 RULE: The system must store the corresponding information
to a rule, specifically: rule action (drop, accept, reject, redirect) the
source and destination port (if any) ip address and destination (if any)
network (if any) protocol, connection status, etc.
5.3. Business rules
RN-001 CREATE A TABLE: It will not be allowed to create a table with a
name that already exists or of a family other than those established by the system
(ip, ip6, nat, raw, arp).
RN-002 DELETE A TABLE: It will not be allowed to delete a table that
contain strings with rules (if strings are empty if you can delete the
table).

5.4. Functional requirements
RF-001: The system must allow a table, chain or rule to be
added or modified at any time respecting business rules.
RF-002: The system must allow export / import of the rule set
tables and strings stored or supplied by the user.
RF-003: In the creation of rules, the system must allow the use of options
to create the rule, this is: choice of source / destination port, source ip
gen / destination, source / destination network, actions on packets (drop, reject, accept, redirect),
protocol used, interface used, etc.
RF-004: The system must show the created rules and be able to carry out actions
about them.
